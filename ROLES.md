# 设备角色说明 & 网卡需求

## 一、各角色作用

| 角色 | 全称 | 提供方 | 功能 |
|------|------|--------|------|
| **CR** | 核心路由器 | 中兴 | 核心网数据转发。AID↔RID 封装/解封装，执行 RID 二维前缀乘积路由。**不部署 Python 代码，仅需配置表项。** |
| **CS** | 控制服务器 | 北交大 | 三合一控制平面：AAA 用户认证 + AID↔RID 映射注册/查询 + 路由标识空间管理。用户注册、凭证验证、移动切换全局传播。 |
| **TS** | 测试服务器 | 北交大 | 业务验证工具：部署 HTTP/FTP/Video 服务供 Host 访问，构建 RID 探测包测试 CR 转发，采集带宽/时延/抖动/丢包率。 |
| **AP** | 接入设备 | 北交大 | 网络边缘代理：将 Host 的传统 IPv4 数据封装为 AID 帧（EtherType 0x1111），代理 Host 向 CS 发起认证，用户上线时通知 CR 并注册映射。 |
| **Host** | 用户终端 | 北交大 | 普通终端：加载 AID 配置文件，发起认证请求，运行 HTTP/FTP/Video 客户端产生业务流量。**本身不需要 AF_PACKET，与 AP 同机时走 loopback。** |

---

## 二、数据流

```
Host ──IPv4──▶ AP ──AID(0x1111)──▶ CR(接入) ──RID(0x2222)──▶ CR(核心) ──RID──▶ CR(核心) ──AID──▶ AP ──IPv4──▶ TS
              ▲                              ▲                    ▲                    ▲
         传统IP→AID封装            AID→RID 封装映射          RID 二维路由         RID→AID 解封装
```

- Host ↔ AP：标准 IP，无需标识网络帧
- AP ↔ CR 接入侧：AID 帧 (EtherType 0x1111)
- CR ↔ CR 核心侧：RID 帧 (EtherType 0x2222)
- CS ↔ CR 管理侧：RID 控制信令 (EtherType 0x2222, DataType=0x01)

## 三、TS 和 AP-1 为什么能共用一张网卡

同一张物理网卡可以通过 AF_PACKET 绑定多个 MAC 地址：

```
Linux 服务器 eth1
├── 进程 TS:  AF_PACKET → MAC=00:1A:2B:3C:4D:02
├── 进程 AP-1: AF_PACKET → MAC=00:04:AB:1F:40:A6
└── eth1 ──── 交换机 #2 VLAN 20
```

原理：AF_PACKET socket 按 MAC 地址收发。两个进程绑在同一张物理网卡但注册了不同的 MAC，交换机在同一个物理端口上学习到两个 MAC 地址，根据帧的 dst MAC 正确转发。Linux 内核也根据 dst MAC 将帧投递给对应的 socket。

---

## 四、网卡需求

TS 和 AP 共处同一接入域（VLAN 20），自然共用一张网卡。CS 在管理交换机上，必须独立一张。AP-1 和 AP-2 在不同接入域，需物理分离或 VLAN tag。

### 方案 A：3 网卡（推荐）

```
eth0 → 交换机 #1 (管理面)  — CS
eth1 → 交换机 #2 (接入面1) — TS + AP-1
eth2 → 交换机 #2 (接入面2) — AP-2
```

### 方案 B：2 网卡（VLAN tag）

```
eth0 → 交换机 #1 (管理面)  — CS
eth1 → 交换机 #2           — TS + AP-1 + AP-2
       eth1 配置 VLAN 子接口:
         eth1.20 — TS + AP-1 (接入面1)
         eth1.30 — AP-2 (接入面2)
```

### 方案 C：1 网卡（极简, 仅验证用）

```
eth0 → 交换机 #2 — 所有角色 (CS + TS + AP-1 + AP-2)
# 控制面和数据面在同一交换机, 仅基础功能验证
```
